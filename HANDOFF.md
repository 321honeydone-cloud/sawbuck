## Hard rules (do not break)
- NO em dashes or en dashes anywhere. Ever. Use commas, periods, or rephrase.
- No semicolons in app copy.
- When offering options, use popup multiple-choice questions.
- PINs are stored readable on purpose (Manny's informed choice).
- Mount-truncation hazard: the Edit tool and bash mount can DIVERGE on this project. After ANY edit, verify on disk via bash (sed/wc). When in doubt, write whole files via bash heredoc, not Edit.
- TYPECHECK GOTCHA (important): in the Linux sandbox, `node_modules/.prisma/client/index.d.ts` and `.next/types/*` get READ-TRUNCATED through the mount, so a plain `npx tsc --noEmit` dies on a parse error in those files BEFORE it checks your code. It will look clean while real type errors slip to the Windows build. Reliable check instead: a temp config that stubs Prisma and skips .next. Recipe (rebuild /tmp/tc each session):
    - `/tmp/tc/prisma-stub.ts`: export a permissive `class PrismaClient` whose models (`catalog`, `user`, `estimate`, `chatMessage`, `inspection`, `userPreset`) are a Delegate with `findMany(args?:any):Promise<any[]>`, `findUnique/findFirst:Promise<any|null>`, `create/update/upsert/delete:Promise<any>`, `count:Promise<number>`.
    - `/tmp/tc/tsconfig.json`: strict, skipLibCheck, moduleResolution bundler, jsx preserve, `types:["node"]`, `typeRoots:["<abs>/node_modules/@types"]`, `baseUrl:"<abs handoff>"`, `paths:{ "@/*":["src/*"], "@prisma/client":["../../../../tmp/tc/prisma-stub.ts"] }`, include only `<abs>/src/**/*.ts(x)`.
    - Run `npx tsc -p /tmp/tc/tsconfig.json | grep "error TS"`. Residual TS7006 on raw `prisma.X.findMany(...).map(...)` rows in untouched files are stub artifacts (real typed client is fine), ignore those. Any error in a file YOU changed is real.
- Launcher runs production build (`npm run build` + `npm run start`), and `npx prisma db push` regenerates the client. Manny must relaunch to see changes; relaunch also reloads the rate book.

## Installing HoneyDone as a desktop app
1. Start the server as usual (double-click "HoneyDone Estimating" / Start HoneyDone.bat). Keep that window open.
2. In Chrome (or Edge) at the app URL, click the install icon in the address bar (monitor with a down-arrow), or menu > Cast, save, and share > Install page as app. Pick Install.
3. It gets its own window, a Start-menu/desktop icon, and the gold-on-black icon. Launch it like any app. The local server still has to be running first (the .bat), since the AI and database are local.
- To change the icon later: drop the real logo PNG in the handoff folder and re-run outputs/make_icon.py, or just tweak the geometry in that script. Re-run, then relaunch.

## HOTFIX 2026-06-23 (app stuck on the loading splash)
Symptom: Sawbuck hung on the boot splash. Cause: `next build` was FAILING, so `npm run build && npm run start` never reached start, the server never answered, and the splash pinged forever. The boot.log (launcher/boot.log) showed the real error:
  src/lib/auth.ts:70 Type error: 'Uint8Array<ArrayBufferLike>' is not assignable to 'BufferSource'
The password-hash code from Workstream 5 passed a salt (a widened typed-array union) into crypto.subtle.deriveBits. Under TS 5.7 the DOM lib wants an ArrayBuffer-backed view. Fixed by casting the Web Crypto args to BufferSource in hashPassword (auth.ts). The pre-existing hmac() uses the same te.encode pattern and was always fine, so only the salt path needed it.
- IMPORTANT LESSON FOR THE SANDBOX TYPECHECK: the stubbed `npx tsc` in the Linux sandbox does NOT reliably catch DOM/Web-Crypto lib-strictness errors, because the TypeScript lib .d.ts files (lib.dom.d.ts, lib.es5.d.ts) read TRUNCATED through the mount, so tsc loads a lenient/incomplete lib. The sandbox check is good for app-logic type errors but can MISS lib typing errors like BufferSource. The real gate is `next build` on Windows. If a future change touches Web Crypto, typed arrays, or other DOM lib types, assume the sandbox may miss it and check launcher/boot.log after the first Windows build.
- If the splash ever hangs again: open launcher/boot.log (or run Start HoneyDone.bat to watch the build live). A "Failed to compile" there is the cause.
